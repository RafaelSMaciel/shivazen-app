 ShivaZen - Regras de Negocio do Banco de Dados
 Schema v3 - Redesign Final
 Data: 2026-04-04
=====================================================================

Este documento descreve as regras de negocio implementadas no banco
de dados da aplicacao ShivaZen (clinica de estetica).

=====================================================================
1. CONTROLE DE ACESSO E USUARIOS
=====================================================================

1.1 PERFIS E FUNCIONALIDADES (RBAC)
- O sistema usa controle de acesso baseado em perfis (Role-Based).
- Cada Perfil (ex: Administrador, Recepcionista) possui N funcionalidades.
- A relacao Perfil <-> Funcionalidade eh N:N via tabela intermediaria
  (perfil_funcionalidade) com restricao UNIQUE (perfil, funcionalidade).
- Cada Usuario tem exatamente 1 perfil (ou nenhum, nullable).

1.2 USUARIO
- Autenticacao via email (campo unico) + senha (hash bcrypt).
- Campo 'ativo' controla se o usuario pode fazer login.
- Relacao 1:1 opcional com Profissional: um usuario pode ser vinculado
  a um profissional para acessar a agenda pessoal.
- DELETE de Perfil eh RESTRICT (nao permite excluir perfil em uso).
- DELETE de Profissional eh SET NULL (usuario mantem acesso admin).

=====================================================================
2. CLIENTES
=====================================================================

2.1 CADASTRO
- Cliente identificado por nome_completo + CPF (unico, opcional).
- Dados de contato: email, telefone, cep, endereco.
- Campo criado_em registra data de cadastro automaticamente.

2.2 CONTROLE DE FALTAS (REGRA DE 3 STRIKES)
- faltas_consecutivas: contador incrementado a cada falta.
- bloqueado_online: ativado automaticamente quando faltas >= 3.
- Cliente bloqueado NAO pode agendar online (apenas por telefone/presencial).
- Ao comparecer a um atendimento, faltas sao resetadas para 0.

=====================================================================
3. PROCEDIMENTOS E PRECOS
=====================================================================

3.1 PROCEDIMENTO
- Cada procedimento tem nome, descricao, duracao em minutos e flag ativo.
- Procedimentos inativos nao aparecem para agendamento.

3.2 VINCULO PROFISSIONAL-PROCEDIMENTO
- Relacao N:N via profissional_procedimento (UNIQUE profissional+procedimento).
- Define quais procedimentos cada profissional pode realizar.

3.3 TABELA DE PRECOS (COM HISTORICO)
- Cada Preco vincula procedimento + profissional (opcional) + valor.
- profissional_id NULL = preco GENERICO do procedimento.
- profissional_id preenchido = preco ESPECIFICO daquele profissional.
- Campo vigente_desde permite historico de precos ao longo do tempo.
- Regra de consulta: buscar preco mais recente com vigente_desde <= hoje.
  Prioridade: preco especifico do profissional > preco generico.

3.4 DISPONIBILIDADE E BLOQUEIOS
- DisponibilidadeProfissional: define horarios de trabalho por dia da semana.
  - dia_semana: 1=Dom, 2=Seg, ..., 7=Sab (CHECK 1-7).
  - SEM unique_together: suporta MULTIPLOS TURNOS por dia (ex: manha e tarde).
  - hora_inicio e hora_fim definem cada janela de atendimento.
- BloqueioAgenda: intervalos em que o profissional NAO atende.
  - profissional_id NULL = bloqueio GERAL (feriado, manutencao).
  - Usado para ferias, reunioes, feriados, etc.

=====================================================================
4. PROMOCOES
=====================================================================

- Promocao pode ser vinculada a um procedimento especifico (ou geral se NULL).
- Campos de desconto: desconto_percentual OU preco_promocional (valor fixo).
- Vigencia controlada por data_inicio e data_fim + flag ativa.
- Propriedade calculada 'esta_vigente': ativa AND dentro do periodo.
- Ao agendar com promocao, o valor_cobrado reflete o desconto e
  valor_original registra o preco cheio para auditoria.

=====================================================================
5. AGENDAMENTO (ATENDIMENTO)
=====================================================================

5.1 CICLO DE VIDA DO ATENDIMENTO
- Status controlado por CHECK constraint:
  AGENDADO -> CONFIRMADO -> REALIZADO
  AGENDADO -> CANCELADO
  AGENDADO -> FALTOU
  CONFIRMADO -> CANCELADO
  CONFIRMADO -> FALTOU

5.2 CAMPOS IMPORTANTES
- cliente_id, profissional_id, procedimento_id: RESTRICT delete (protege dados).
- promocao_id: SET NULL se promocao removida.
- reagendado_de_id: self-reference para rastrear reagendamentos.
  - Permite encadear: original -> reagendamento1 -> reagendamento2.
- data_hora_inicio / data_hora_fim: janela do atendimento.
- valor_cobrado: valor efetivamente cobrado (com desconto se houver).
- valor_original: valor cheio antes de desconto (auditoria).
- descricao_preco: texto livre justificando o preco.

5.3 REGRAS DE AGENDAMENTO
- Verificacao de conflito: nao permite 2 atendimentos no mesmo horario
  para o mesmo profissional (verificado na aplicacao).
- Verificacao de bloqueio: horario nao pode estar em periodo bloqueado.
- Intervalo padrao de slots: 30 minutos.

=====================================================================
6. PRONTUARIO (SISTEMA HIBRIDO)
=====================================================================

6.1 ANAMNESE PERMANENTE (Prontuario)
- Relacao 1:1 com Cliente (UNIQUE cliente_id).
- Campos de saude persistentes: alergias, contraindicacoes,
  historico_saude, medicamentos_uso, observacoes_gerais.
- Atualizado pelo profissional a qualquer momento.
- atualizado_em: auto-update a cada salvamento.

6.2 PERGUNTAS DINAMICAS
- ProntuarioPergunta: banco de perguntas configuravel.
  - tipo_resposta: TEXTO, BOOLEAN ou SELECAO (CHECK constraint).
  - Flag 'ativa' permite desativar perguntas sem perder historico.
- ProntuarioResposta: respostas do cliente vinculadas ao prontuario.
  - UNIQUE (prontuario, pergunta): cada cliente responde 1x por pergunta.

6.3 ANOTACOES POR SESSAO (AnotacaoSessao)
- Observacoes clinicas ESPECIFICAS de cada atendimento.
- Vinculado a: atendimento + usuario (profissional que anotou).
- Permite multiplas anotacoes por atendimento (evolucao do caso).
- Diferente do prontuario que eh permanente — anotacoes sao por sessao.

=====================================================================
7. TERMOS DE CONSENTIMENTO (VERSIONADOS)
=====================================================================

7.1 VERSAO DO TERMO
- Dois tipos: LGPD (privacidade) ou PROCEDIMENTO (CHECK constraint).
- Quando tipo=LGPD: procedimento_id eh NULL.
- Quando tipo=PROCEDIMENTO: vinculado ao procedimento especifico.
- Versionamento: campo versao (ex: '1.0', '2.1') + vigente_desde.
- Flag 'ativa': apenas uma versao ativa por tipo/procedimento.

7.2 ACEITE DE PRIVACIDADE (LGPD)
- Cliente assina 1x por versao do termo LGPD.
- UNIQUE (cliente, versao_termo): impede duplicata.
- Registra IP do aceite + timestamp.
- Se nova versao publicada, cliente precisa assinar novamente.

7.3 ASSINATURA DE TERMO DE PROCEDIMENTO
- Cliente assina 1x por versao de termo de procedimento.
- UNIQUE (cliente, versao_termo): persiste entre sessoes.
- atendimento_id: registra qual atendimento gerou a assinatura.
- Nao precisa reassinar a cada sessao do mesmo procedimento.
- Nova versao do termo exige nova assinatura.

=====================================================================
8. NOTIFICACOES
=====================================================================

- Vinculada ao atendimento (CASCADE delete).
- Tipos: LEMBRETE, CONFIRMACAO, CANCELAMENTO, NPS.
- Canais: WHATSAPP (principal), SMS, EMAIL.
- Status de envio: PENDENTE, ENVIADO, FALHOU.
- Token unico: usado em links de confirmacao/cancelamento.
- Resposta do cliente via link: CONFIRMOU ou CANCELOU.
- Workflow: notificacao enviada -> cliente clica no link -> resposta registrada
  -> status do atendimento atualizado -> admin notificado.

=====================================================================
9. AVALIACAO NPS
=====================================================================

- Relacao 1:1 com Atendimento (UNIQUE atendimento_id).
- Nota: escala NPS real 0-10 (CHECK constraint 0 <= nota <= 10).
  - Detratores: 0-6
  - Neutros: 7-8
  - Promotores: 9-10
- Comentario opcional para feedback qualitativo.
- alerta_enviado: flag para evitar alertas duplicados ao admin.
- Enviado automaticamente via WhatsApp apos atendimento REALIZADO.

=====================================================================
10. PACOTES DE SESSOES
=====================================================================

10.1 ESTRUTURA DO PACOTE
- Pacote: template com nome, preco_total, validade_meses.
- ItemPacote: define procedimentos e quantidade de sessoes incluidas.
  - UNIQUE (pacote, procedimento): um procedimento por pacote.

10.2 COMPRA DO PACOTE (PacoteCliente)
- Registra compra pelo cliente: valor_pago, data_expiracao.
- Status: ATIVO, FINALIZADO, CANCELADO, EXPIRADO (CHECK constraint).
- data_expiracao calculada automaticamente: data_compra + validade_meses.

10.3 CONTROLE DE SESSOES (SessaoPacote)
- Vincula atendimento ao pacote comprado (1:1 com Atendimento).
- Permite rastrear quantas sessoes de cada procedimento foram usadas.
- Quando todas as sessoes de todos os itens forem realizadas,
  o pacote eh automaticamente marcado como FINALIZADO.

=====================================================================
11. LISTA DE ESPERA
=====================================================================

- Cliente solicita vaga quando horario desejado nao esta disponivel.
- Campos: procedimento, profissional_desejado (opcional), data, turno.
- Turno: MANHA, TARDE, NOITE (CHECK constraint, nullable).
- Quando vaga abre: sistema notifica e gera token_reserva com expira_em.
- Cliente tem tempo limitado para confirmar reserva via token.
- Flag 'notificado' evita notificacoes duplicadas.

=====================================================================
12. AUDITORIA E SISTEMA
=====================================================================

12.1 LOG DE AUDITORIA
- Registra acoes criticas dos usuarios.
- Campos: acao (texto), tabela_afetada, id_registro, detalhes (JSON).
- usuario_id SET NULL se usuario deletado (mantem log).
- Ordenado por criado_em para timeline de eventos.

12.2 CONFIGURACAO DO SISTEMA
- Tabela chave-valor para configuracoes dinamicas.
- Chave unica (ex: 'whatsapp_admin', 'horario_funcionamento').
- Permite alterar comportamento sem deploy.

12.3 CODIGO DE VERIFICACAO
- Usado para autenticacao por SMS/WhatsApp (passwordless).
- Codigo de 6 digitos com validade de 10 minutos.
- Flag 'usado' impede reutilizacao.
- Validacao: nao usado AND (agora - criado_em) < 600 segundos.

=====================================================================
13. REGRAS DE INTEGRIDADE REFERENCIAL
=====================================================================

RESTRICT (impede delete se ha dependentes):
- Perfil -> Usuario
- Procedimento -> Atendimento
- Profissional -> Atendimento
- Cliente -> Atendimento
- Pacote -> PacoteCliente
- Atendimento -> SessaoPacote
- VersaoTermo -> AceitePrivacidade
- VersaoTermo -> AssinaturaTermoProcedimento
- ProntuarioPergunta -> ProntuarioResposta

CASCADE (deleta dependentes junto):
- Perfil -> PerfilFuncionalidade
- Profissional -> DisponibilidadeProfissional, BloqueioAgenda, ProfissionalProcedimento, Preco
- Procedimento -> ProfissionalProcedimento, Preco, Promocao, VersaoTermo
- Cliente -> Prontuario, AceitePrivacidade, AssinaturaTermoProcedimento, PacoteCliente, ListaEspera
- Atendimento -> AnotacaoSessao, Notificacao, AvaliacaoNPS
- Prontuario -> ProntuarioResposta
- Pacote -> ItemPacote
- PacoteCliente -> SessaoPacote

SET NULL (mantem registro, remove referencia):
- Profissional -> Usuario.profissional_id
- Promocao -> Atendimento.promocao_id
- Atendimento -> Atendimento.reagendado_de_id
- Usuario -> AnotacaoSessao.usuario_id
- Atendimento -> AssinaturaTermoProcedimento.atendimento_id
- Usuario -> LogAuditoria.usuario_id
- Profissional -> ListaEspera.profissional_desejado_id

=====================================================================
14. CHECK CONSTRAINTS
=====================================================================

- chk_disponibilidade_dia_semana: dia_semana BETWEEN 1 AND 7
- chk_atendimento_status: status IN ('AGENDADO','CONFIRMADO','REALIZADO','CANCELADO','FALTOU')
- chk_prontuario_pergunta_tipo: tipo_resposta IN ('TEXTO','BOOLEAN','SELECAO')
- chk_versao_termo_tipo: tipo IN ('LGPD','PROCEDIMENTO')
- chk_avaliacao_nps_nota: nota BETWEEN 0 AND 10
- chk_pacote_cliente_status: status IN ('ATIVO','FINALIZADO','CANCELADO','EXPIRADO')
- chk_lista_espera_turno: turno_desejado IS NULL OR IN ('MANHA','TARDE','NOITE')

=====================================================================
15. UNIQUE CONSTRAINTS
=====================================================================

- funcionalidade.nome
- perfil.nome
- perfil_funcionalidade (perfil_id, funcionalidade_id)
- usuario.email
- usuario.profissional_id
- cliente.cpf
- profissional_procedimento (profissional_id, procedimento_id)
- notificacao.token
- prontuario.cliente_id (1:1)
- prontuario_resposta (prontuario_id, pergunta_id)
- avaliacao_nps.atendimento_id (1:1)
- sessao_pacote.atendimento_id (1:1)
- item_pacote (pacote_id, procedimento_id)
- aceite_privacidade (cliente_id, versao_termo_id)
- assinatura_termo_procedimento (cliente_id, versao_termo_id)
- configuracao_sistema.chave
